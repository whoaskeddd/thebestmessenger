# План работ по бекенду (FastAPI) — HR + мессенджер

Цель: сделать работающий бекенд для интегрированной HR‑системы с внутренним мессенджером (см. корневой `plan.md`), придерживаясь текущей структуры `backend/app` и правил из `AGENTS.md` (Vertical Slice + Repository Pattern).

## 0) Решения и вопросы (чтобы зафиксировать план)

### 0.1. Уже зафиксировано
- **Python**: используем **3.12** (нужно привести `backend/pyproject.toml` к `requires-python >=3.12`).
- **БД**: **PostgreSQL в Docker** (версия не принципиальна; используем актуальную в `docker-compose`).
- **DB драйвер**: `asyncpg` (async stack).
- **Аутентификация**: email + password + JWT (Bearer).
- **Роли (MVP)**: `admin`, `hr`, `employee` (список может расширяться).
- **Оргструктура**: у сотрудника может быть **несколько отделов**.
- **Заявки (MVP типы)**: отпуск, отгул, больничный.
- **Объявления**: таргетинг по отделам.
- **Мессенджер**: личные и групповые чаты; сообщения + голосовые (ГС).
- **Dev запуск**: `docker-compose` для всего (API + Postgres и зависимости).

### 0.2. Принятые решения (чтобы не блокироваться)
1) **JWT детали (MVP)**:
   - Access token: **15 минут**.
   - Refresh token: **30 дней**, **ротация refresh** при обновлении.
   - Храним **хеши refresh токенов** в БД (revocation/rotation); при reuse — инвалидируем сессию.
   - Инвалидация при смене пароля/увольнении: поле **`token_version`** у пользователя + принудительный logout (старые токены перестают проходить).
2) **Сотрудники и пользователи (MVP)**:
   - Связь `user` ↔ `employee`: **1:1**, но допускаем `employee` **без** `user` (не активирован/ещё не выдана учётка).
3) **Workflow заявок (MVP)**:
   - Согласует **только HR** (роли `manager` нет в MVP).
   - При отказе обязателен **комментарий**.
   - “Второй апрувер/замещающий” — **не** в MVP.
4) **HR‑задачи (MVP)**:
   - Отдельная сущность `hr_task`, может ссылаться на `announcement` (опционально), но живёт отдельно.
5) **Голосовые сообщения (MVP)**:
   - Хранение файлов: **только локально** (внутри контейнера API) через volume в `docker-compose`.
   - Ограничения: до **10 MB** и до **120 секунд** на сообщение.
7) **Объявления по отделам (MVP)**:
   - Объявление может таргетиться на **несколько отделов**.
   - Поддерживаем “**всем**” флагом `is_global=true` (или `target_departments` пустой + флаг для явности).

### 0.3. Зафиксировано по мессенджеру (MVP)
- Реал‑тайм статусы: **отправлено**, **доставлено**, **прочитано** (без “печатает”).

## 1) Область работ (MVP)

### Входит
- JWT auth + RBAC.
- Сотрудники/профили + отделы.
- Заявки на отпуск/отгулы + простой workflow согласования + история.
- Объявления + HR‑задачи + отметки “прочитал/выполнил”.
- Мессенджер: чаты/сообщения + WebSocket (реал‑тайм) + хранение в БД.

### Не входит (пока)
- Полнотекстовый поиск, advanced analytics.
- Push‑уведомления на мобилках (только backend hooks/эндпоинты).
- Загрузки файлов, вирус‑скан, CDN (если не подтвердим в вопросах).
- Сложные орг‑матрицы (если не подтвердим).

## 2) Архитектура и договорённости

Опираемся на `AGENTS.md`:
- Вертикальные слайсы по фичам в `api/domain/infrastructure/schemas`.
- Repository interfaces в `domain`, реализации в `infrastructure`, внедрение через DI на уровне `api/core`.
- `domain` не импортирует FastAPI/ORM модели/DB session; `api` не содержит бизнес‑логики.
- Async persistence: репозитории работают поверх `AsyncSession`.

Предлагаемые слайсы (папки):
- `auth` — пользователи, токены, роли/права
- `employees` — сотрудники, профили, отделы
- `leave_requests` — заявки, статусы, история, approvals
- `announcements` — объявления, “прочитал”
- `hr_tasks` — задачи, “выполнил”
- `messenger` — чаты, участники, сообщения, websocket

## 3) Валидация (допущения)

Пока в репо нет тестов/скриптов, предлагаю такие команды (уточним/поправим по факту):
- `uv sync`
- `uv run python -m compileall backend/app`
- `uv run pytest -q` (когда появятся тесты)

## 5) Критерии готовности MVP (Definition of Done)
- Документированные эндпоинты (OpenAPI) + понятный способ локального запуска.
- Работает auth + RBAC.
- CRUD сотрудников/отделов.
- Workflow заявок работает и логируется.
- Объявления/задачи работают.
- Мессенджер работает: REST + WebSocket + хранение сообщений.
- Есть тесты: доменные (use-case) + хотя бы базовые интеграционные для persistence.
